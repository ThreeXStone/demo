# 任务复盘：总结经验教训

## 任务目标

对刚刚完成的任务进行复盘，提取错误、弯路和经验教训，更新到 `CLAUDE.md` 的踩坑章节，避免下次重蹈覆辙。

---

## 执行步骤

### 第一步：回顾本次任务过程

梳理刚才的完整执行过程，重点关注以下几类问题：

- 用错了命令或工具（如用 npm 代替 bun）
- 路径、端口、服务名写错
- 配置文件格式错误
- 遗漏了必要的步骤（如忘记在 Module 里注册）
- 被用户纠正过的地方
- 走了弯路后才找到正确方向的地方

### 第二步：整理成结构化记录

每条经验按以下格式整理：

```
- ❌ 错误做法：`<具体命令、配置或操作>`
  ✅ 正确做法：`<具体命令、配置或操作>`
  📌 原因：<一句话说明为什么，帮助下次理解而不只是死记>
```

只记录**有价值的教训**，跳过以下内容：
- 正常执行、没出问题的步骤
- 用户需求变更导致的调整（不是错误）
- 过于显而易见、不会再犯的低级笔误

### 第三步：更新 CLAUDE.md

打开项目根目录的 `CLAUDE.md`：

- 如果已有 `## 踩过的坑` 章节，将新内容**追加**到末尾，不要覆盖已有记录
- 如果没有该章节，创建它并写入内容
- 每组记录前加上本次任务的简短标题作为注释，方便以后按任务查找：

```markdown
## 踩过的坑

<!-- [任务名称，如：LangChain 提示模板] -->
- ❌ 错误做法：...
  ✅ 正确做法：...
  📌 原因：...
```

### 第四步：输出复盘摘要

在对话里简要输出本次复盘结果：
- 本次任务名称
- 共记录了几条教训
- 最值得注意的一条是什么

---

## 注意事项

- 如果本次任务执行顺利、没有犯错，如实说明"本次无新增踩坑记录"，不要强行编造
- 描述要具体，避免模糊表达（如"注意配置文件"不如"ConfigModule 必须在 app.module.ts 顶部 import"）
- 原因要解释清楚"为什么"，而不只是"是什么"
